# ADR-001 — Persistence & State Model

- **Status:** Accepted (Phase 0, ratified)
- **Date:** 2026-06-26
- **Deciders:** Architecture Review Board
- **Supersedes / relates:** `blueprint/v2/09_ADR_BACKLOG.md` ADR-001; closes gap G1, G5, G10; informs ADR-002/003/004 and all of Phase 2.
- **Affected Action Points:** AP-001 (this decision), AP-101/112 (object schemas), AP-201/202/203/204 (substrate), AP-503 (recovery), AP-604 (audit).

---

## 1. Context

Three cross-cutting models describe persistence, each authoritative in its own
document but silent on how they relate:

- **Event Model (`23`)** mandates immutable events, a delivery layer, **replay**,
  and event-driven coordination. "Operational health is derived from event
  patterns"; "completed events become operational knowledge."
- **State Model (`24`)** mandates one unified lifecycle, transition guards, and
  **exactly one Event per transition** — but never says whether current state is
  *derived from* the event log or *stored independently*.
- **Checkpoint Model (`25`)** mandates **reference-not-copy snapshots** from
  which Recovery resumes ("recover from state, never from operator intent").

The architecture also makes promises that depend on the answer: Recovery resumes
from checkpoints (`19`), Knowledge is mined from validated events (`10`, `26`),
Governance keeps an immutable audit (`12`), and Supervision derives health from
event streams (`09`). Every one of these assumes a durable, ordered,
reconstructable operational history.

## 2. Problem Statement

Nexus has **three candidate sources of truth** for "where is this operation
now": the event log, the state store, and the checkpoint store. If they persist
independently they can diverge, and every promise above (replay, recovery,
audit, knowledge) becomes unreliable. The architecture never decided whether the
platform is **event-sourced, state-based, or hybrid**. This is the single
highest-leverage unmade decision: all of Phase 2 and the correctness of Recovery
and Audit depend on it.

A secondary contradiction (G10): the Resource Model (`22`) defines its own
availability state machine while the State Model (`24`) lists Resource as a
stateful object on the *generic* lifecycle — two state machines for one object.

## 3. Decision

**Nexus is event-sourced with materialized projections. The append-only Event
Log is the single authoritative source of operational truth. Current State and
Checkpoints are derived artifacts, never independent sources of truth.**

Concretely:

1. **Event Log is authoritative.** Every operational fact is an immutable,
   ordered, append-only Event. Nothing that is not in the log is true. The log
   is the audit record, the replay source, and the substrate Knowledge mines.

2. **State is a projection (read model).** The current State of any object
   (Goal, Plan, Work Package, Execution Session, Artifact, Resource, Validation,
   Recovery, Knowledge Entry) is a deterministic fold over its events. State is
   stored as a **materialized projection** for fast reads and transition-guard
   evaluation, but it is *rebuildable from the log at any time* and is never the
   source of truth. The rule "exactly one Event per transition" (`24`) is the
   **write path**; the projection is the read path.

3. **Checkpoints are materialized snapshots, not a third store.** A Checkpoint
   is a reference-bearing snapshot of execution-relevant projected state at a
   point in the log (it records the **log position / sequence** it corresponds
   to). Recovery restores the nearest valid Checkpoint and **replays the event
   tail** from that position. Checkpoints exist only to bound replay cost; they
   are derived from the log and can be regenerated.

4. **Resource availability is one projection in the unified State Model**
   (resolves G10). The Resource availability vocabulary (Available/Busy/
   Reserved/Offline/Maintenance/Failed) is expressed as a *specialized
   projection* of the generic lifecycle, not a separate independent machine.
   Allocation transitions emit events like every other transition.

5. **Idempotency is mandatory on the write path.** Because delivery is
   at-least-once (`23`), every event carries a stable identity and every
   projection applies events idempotently (dedup by event id + sequence). This
   is the platform-wide idempotency contract (AP-202), not a per-consumer
   concern.

6. **Determinism of projection.** Folding the same event sequence always yields
   the same state. Projections may not call non-deterministic functions during
   replay; any non-deterministic value (LLM output, human decision, clock) is
   captured *as recorded event data*, never recomputed (ties to ADR-004 §
   determinism boundaries).

## 4. Alternatives Considered

**A. Pure state-based (CRUD) with an event side-channel.** Current state is the
primary store; events are emitted for observability. *Rejected:* the event log
becomes a lossy side-channel that can disagree with state, breaking replay,
recovery correctness, and audit integrity. Two sources of truth is the exact
failure this ADR exists to prevent. Simpler reads do not justify
unreconstructable history.

**B. Pure event sourcing with no stored projection.** State is always folded on
demand; no materialized read model. *Rejected (as a sole approach):* read and
guard-evaluation cost grows with stream length; long-running operations make
hot-path transition checks expensive. Snapshots/projections are required in
practice — which is precisely the hybrid we chose.

**C. Three independent stores (status quo of the docs).** Event, State, and
Checkpoint each persist independently. *Rejected:* maximal divergence risk; no
defined reconciliation; the architecture's replay/recovery/audit promises cannot
be guaranteed.

**D. Event sourcing with periodic full-state snapshots (no checkpoint
semantics).** *Partially adopted:* our Checkpoints *are* snapshots, but they
carry recovery-specific metadata (graph position, artifact references, policy
compatibility) the architecture already requires (`25`), so we keep the
Checkpoint concept rather than a generic snapshot.

## 5. Trade-offs

- **Gain:** one source of truth; native audit, replay, recovery, and knowledge
  mining; no divergence; deterministic reconstruction; clean resolution of the
  Resource dual-state contradiction.
- **Cost:** higher operational complexity (projection rebuild, snapshotting,
  event versioning/upcasting, idempotency discipline); engineers must think in
  events, not rows; eventual consistency between the log and projections must be
  reasoned about explicitly.
- **Accepted because:** the architecture *already* committed to replay,
  recover-from-state, immutable audit, and event-derived knowledge. Those are
  event sourcing in all but name. Choosing state-based would contradict commit-
  ments already made across four documents. We optimize for long-term
  correctness over implementation simplicity, per the mandate.

## 6. Consequences

- **Phase 2** builds an **append-only event store** (AP-201) as the spine, a
  **projection engine** (AP-203) that materializes State from the log, and a
  **checkpoint store** (AP-204) that snapshots projections with a log position.
- **Idempotency/correlation** (AP-202) is foundational, not optional.
- **Recovery** (AP-503) becomes "restore nearest checkpoint → replay tail,"
  giving exact, evidence-preserving resume.
- **Audit** (AP-604) is the log itself plus governance decision events; no
  separate audit store can disagree with operations.
- **Knowledge** (AP-602) ingests only validated events/outcomes; the log makes
  provenance trivial.
- **Contracts** (`contracts/event.md`, `state` semantics, `checkpoint.md`) must
  state that State and Checkpoint are *derived*; Event is *authoritative*.
- **Schema versioning** (AP-101) must support **event upcasting** (old events
  remain replayable forever) — event schemas are append-only and backward-
  compatible.

## 7. Risks

- **R-01 (persistence) retired into execution risk:** projection/log divergence
  if idempotency or ordering is wrong. *Mitigation:* AP-202 idempotency;
  causal ordering per correlation id; replay-equivalence tests (project twice →
  identical state).
- **Replay cost** on long streams. *Mitigation:* checkpoint cadence policy
  (Execution Strategy), snapshot-and-truncate-replay-window.
- **Event schema evolution** breaking old replays. *Mitigation:* append-only,
  backward-compatible event schemas + upcasting; never mutate a published event
  type.
- **Non-deterministic replay** if recorded outputs are recomputed. *Mitigation:*
  capture-as-data rule (§3.6), enforced by ADR-004 determinism boundaries.

## 8. Validation Strategy

- **Replay equivalence test:** fold an event stream twice → byte-identical
  projection (AP-203).
- **Recover-and-resume test:** kill mid-execution, restore nearest checkpoint,
  replay tail → operation continues with no evidence loss and identical final
  state vs. the uninterrupted run (AP-204, AP-503).
- **Idempotency property test:** duplicate/out-of-order delivery causes no
  duplicate state change (AP-202).
- **Audit-completeness test:** every state transition appears exactly once in
  the log; the log fully reconstructs the operation (AP-604).
- Gate: Phase 2 validation gate in `blueprint/v2/10_VALIDATION_STRATEGY.md`.

## 9. Migration Considerations

- **Nexus v1** uses SQLAlchemy/aiosqlite with a transactional outbox
  (`SystemOutboxRecord`) and APScheduler — effectively state-based with an
  outbound event channel. The outbox is a natural seam: v2 promotes the **event
  log to authoritative** and demotes relational tables to **projections**.
- Migration is incremental: introduce the event store and dual-write (state +
  event) behind the existing services; flip reads to projections; then make the
  log authoritative and rebuild projections from it. v1 operational guarantees
  (governance fail-closed, approval gates) are preserved because they become
  policy/governance *events* in the log.
- No big-bang rewrite; the v1→v2 path is "outbox becomes the log."

## 10. Future Evolution

- Distributed event bus / partitioned log (the docs' deferred "distributed event
  buses") — the authoritative-log model is partition-friendly when partitioned
  by correlation id.
- Snapshot compaction and cold-storage tiering for old log segments.
- Event-stream analytics for predictive Supervision and adaptive planning.
- These remain compatible: nothing here assumes a single-node store, only an
  ordered, durable, append-only log per operation.

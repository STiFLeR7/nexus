# ADR-002 — Registry Architecture

- **Status:** Accepted (Phase 0, ratified)
- **Date:** 2026-06-26
- **Deciders:** Architecture Review Board
- **Relates:** `blueprint/v2/09_ADR_BACKLOG.md` ADR-002; closes gap G2, contributes to G10; depends on ADR-001 (state is a projection); informs `contracts/capability.md`, `contracts/resource.md`, `contracts/skill.md`, `contracts/policy.md`.
- **Affected Action Points:** AP-002 (this decision), AP-108/109 (schemas), AP-207 (Harness SDK + registry), AP-304 (capability resolution), AP-402 (allocation).

---

## 1. Context

Six registries are named or implied across the architecture: **Capability**
(`21`), **Resource** (`22`), **Runtime** (`15`), **Harness** (`11`), **Skill**
(`06`), and **Policy** (`20`). Four of them — Harness, Runtime, Resource,
Capability — define **near-identical registration field sets** (Identity,
Capabilities, Availability, Health, Version, Provider). Provider/Availability/
Health appear in **both** the Capability Registry and the Resource Registry,
creating dual sources of truth. The Harness (`11`) is explicitly the *single
integration boundary for all external systems* (Runtime, Context, Knowledge,
Validation, Communication, Governance, Observability categories).

## 2. Problem Statement

Multiple registries with overlapping responsibilities mean:
- No single authority for "is provider X available/healthy right now?"
  (duplicated in Capability and Resource registries).
- The **Runtime Registry** and the **Harness Registry** describe the same thing
  (a Runtime is one Harness category), yet exist as separate concepts.
- **Capability resolution** (capability → providers) is described in the
  Capability Model, which simultaneously disclaims provider selection — so the
  resolution step has no clear owner.
- The word "Resource" conflates two distinct ideas: (a) external capability
  *providers* (Claude Code, GitHub) and (b) allocatable *quotas* (compute,
  budget, human time).

## 3. Decision

**Nexus has FOUR authoritative registries with non-overlapping
responsibilities. The Runtime Registry is subsumed by the Harness Registry. The
Resource Registry is demoted from a discovery registry to an allocation
projection.**

### The four authoritative registries

1. **Capability Registry** — *abstract "what can be done."*
   - Owns: Capability *definitions* (Identifier, Name, Description, Version,
     Category, typed Inputs/Outputs, Constraints, Dependencies).
   - Holds **no** provider, availability, or health state.
   - Consumers: Planning (reasons in capabilities), Capability Resolution.
   - Lifecycle: definitions are versioned, append-only, provider-independent.

2. **Harness Registry** — *the single registry of integration boundaries
   (subsumes Runtime Registry).*
   - Owns: every concrete provider as a Harness (category ∈ {Runtime, Context,
     Knowledge, Validation, Communication, Governance, Observability}), with the
     Harness common contract (Identity, **Advertised Capabilities**, Availability,
     **Health**, Configuration, Authentication, Operations, Events, Errors,
     Metrics, Version).
   - **Sole source of truth for provider Availability and Health.**
   - Lookup responsibility: capability discovery — "Need → Capability → Search
     Registry → Available Harnesses."
   - Consumers: Capability Resolution, Orchestration (runtime assignment),
     Supervision (health/events).
   - A **Runtime is a Harness of category Runtime.** There is no separate Runtime
     Registry.

3. **Skill Registry** — *reusable operational procedures.*
   - Owns: Skill definitions, versions, composition references, required context.
   - Distinct from Capability (Skills are *procedures*; Capabilities are
     *functionality*) and from Harness (Skills are not integration boundaries).
   - Consumers: Skill Selection, Work Packaging.

4. **Policy Registry** — *governance and operational rules.*
   - Owned by the Policy Engine (ADR-004). Holds versioned Policy objects.
   - Distinct kind of artifact (rules, not providers/procedures).
   - Consumers: Policy Engine evaluation, Governance.

### Resource is an allocation projection, not a discovery registry

- The **Resource object** is retained (an allocatable instance), but its
  *registry/discovery* duties move to the Harness Registry for harness-backed
  resources.
- **Resource allocation state** (Allocated/Reserved/Released) is a **projection
  in the unified State Model** (ADR-001), owned operationally by **Orchestration**
  (the sole allocator).
- Non-harness allocatable quotas (compute, budget, human-time windows) live in an
  **Infrastructure Resource catalog** — a thin allocation ledger, **not** a
  capability-discovery registry, and it never duplicates Harness health/availability.

### Field ownership (single-owner table)

| Field | Owner |
|-------|-------|
| Capability definition / typed I/O / version | Capability Registry |
| Provider identity, advertised capabilities | Harness Registry |
| Provider Availability, Health, Auth, Config | **Harness Registry (sole)** |
| Allocation state (Allocated/Reserved/Released) | State projection (Orchestration) |
| Skill definition / procedure / composition | Skill Registry |
| Policy definition / version | Policy Registry |

### Capability Resolution ownership

Capability Resolution (capability → candidate providers) is a **read operation
over the Capability Registry ∪ Harness Registry**, performed in Phase 3
(AP-304). It returns *candidates*. **Selection and allocation are Orchestration's**
(AP-402). Resolution never selects a runtime; allocation never re-discovers
capabilities.

## 4. Alternatives Considered

**A. Keep all six registries as-is.** *Rejected:* perpetuates the dual source of
truth for provider health/availability and the Runtime/Harness duplication.

**B. Collapse everything into one universal registry.** *Rejected:* over-merging
destroys useful, real distinctions (abstract capability vs. concrete provider vs.
procedure vs. rule). A single registry would re-create the field-ownership
confusion internally.

**C. Keep Resource Registry as a peer discovery registry.** *Rejected:* it would
continue to duplicate Harness health/availability. The two real needs —
*discovery* (Harness) and *allocation accounting* (ledger/projection) — are
cleanly separated instead.

**D. Subsume Skills into Capabilities.** *Rejected:* Skills compose and version
as procedures independently of capabilities and are selected by different
criteria (context/evidence). Conflating them couples procedure evolution to
capability typing.

## 5. Trade-offs

- **Gain:** one owner per field; one place to ask "is this provider available?";
  Runtime/Harness duplication eliminated; resolution vs. selection vs. allocation
  cleanly separated.
- **Cost:** the Resource Model doc (`22`) must be re-read through this lens
  (Resource = allocatable instance + state projection, not a discovery registry);
  some intuitive "look it up in the resource registry" phrasing is retired.
- **Accepted because:** eliminating dual sources of truth for health/availability
  is worth the conceptual adjustment; the Harness already *is* the unifying
  integration abstraction the architecture introduced.

## 6. Consequences

- **AP-207 (Harness SDK + registry)** becomes the integration spine and the sole
  provider-state authority; **AP-403/404** runtime adapters are Harness-category
  implementations.
- **AP-108 (Capability schema)** carries no provider/health fields; **AP-109
  (Resource + Harness)** reconciles Resource state into the State Model.
- **AP-304 (Capability Resolution)** is a read across Capability+Harness;
  **AP-402 (allocation)** owns selection/allocation state.
- **Contracts:** `capability.md` (abstract only), `resource.md` (instance +
  allocation state, references Harness for harness-backed resources). The Harness
  contract is documented in the Harness SDK spec; no separate Runtime registry
  contract.

## 7. Risks

- **Over-centralizing on the Harness Registry** makes it a hot path. *Mitigation:*
  health/availability are cached projections updated by harness events (ADR-001);
  reads do not block on live probes.
- **Migration friction** from v1's "Runtime Registry." *Mitigation:* §9.
- **Harness abstraction leakage** across very different categories. *Mitigation:*
  category-specific operations within the common contract; conformance tests per
  category (AP-207).

## 8. Validation Strategy

- **Field-ownership conformance test:** no field (Provider/Availability/Health/
  Version) resolves to two registries (AP-002 → AP-108/109).
- **Discovery test:** a registered Harness is discoverable by capability and
  reports health/availability from exactly one place (AP-207).
- **Resolution/selection boundary test:** Capability Resolution returns
  candidates without selecting a runtime; allocation sets state without
  re-resolving (AP-304/402).
- **Runtime-as-Harness test:** a runtime adapter registers and is discovered
  through the Harness Registry with no separate runtime registry (AP-403).

## 9. Migration Considerations

- **v1 Runtime Registry** (`runtime_registry`, ids "claude"/"gemini"/"nexus")
  maps directly to **Harness Registry, category Runtime**. The v1
  `BaseRuntimeAdapter`/`CLIRuntimeAdapter`/`AgentRuntimeAdapter` become Runtime
  Harness implementations of the common contract.
- v1 has no Capability or Harness abstraction; introduce the Capability Registry
  first (definitions), then wrap existing runtimes as Harnesses, then route
  discovery through it.
- v1 channel/communication and email/SMTP adapters become **Communication
  Harnesses**; the research/context fetchers become **Context Harnesses** —
  unifying integration under one boundary incrementally.

## 10. Future Evolution

- Distributed / remote / sandboxed Harnesses and an organization-wide Harness
  registry (the docs' deferred items) fit unchanged — they are more Harness
  instances under the same contract.
- Dynamic capability negotiation and capability composition operate over the
  Capability Registry without touching provider state.
- A future capability **trust/verification** layer (currently self-reported) can
  attach to the Harness Registry as health/attestation metadata without changing
  ownership.

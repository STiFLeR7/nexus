# ADR-003 — Canonical Object Model

- **Status:** Accepted (Phase 0, ratified)
- **Date:** 2026-06-26
- **Deciders:** Architecture Review Board
- **Relates:** `blueprint/v2/09_ADR_BACKLOG.md` ADR-003 (+ADR-008/009); closes gaps G3, G4, G5, G6, G7, G9, G11; depends on ADR-001; informs every file in `contracts/`.
- **Affected Action Points:** AP-003 (this decision), AP-007 (glossary/index), AP-101…AP-112 (schemas), AP-301 (intent), AP-307 (graph).

---

## 1. Context

The Object Model (`02`) is the platform's shared language: "every subsystem must
consume and produce these objects." But several objects are defined
inconsistently across documents, and the universal input is not defined at all.
Phase 1 cannot produce one authoritative schema per object until these are
reconciled.

## 2. Problem Statement

Seven concrete contradictions block the canonical object model:

- **G7 — Goal undefined.** The universal pipeline input has no structural
  definition anywhere.
- **G3 — Work Package defined three times** with different field sets (Planning
  `04`, Work Packages `05`, referenced in `14`).
- **G4 — Execution Graph containment.** Nested inside Plan (`02`) vs. sibling
  output (`01/04/18`); plus a redundant separate "Dependency Graph" (`04`).
- **G5 — Observation dual ownership.** Execution "produces Observations" (`08`)
  vs. Supervision owns Observation (`02/09`).
- **G6 — Intent Resolution == Executive Intelligence.** Same layer, two names;
  richer Goal metadata in `16` not reflected in `02`.
- **G9 — Validation enum drift.** Result set vs. state set name the same concept
  differently ("Requires Review" vs. "Waiting Human Review").
- **G11 — Artifact dual vocabulary** (Status vs. Lifecycle) and an embedded
  Evidence field overlapping the standalone Evidence object.

## 3. Decision

### 3.1 Goal is defined (G7)

A **Goal** is a normalized desired outcome plus metadata, never a procedure.
Required: `identity`, `outcome` (declarative), `domain`, `priority`,
`confidence`, `constraints`, `scope` (included/excluded). Optional:
`clarifications`, `source`, `correlation`. **Goal Metadata** (domain/priority/
confidence/clarifications produced by Intent Resolution) lives **inside the Goal
object**, not as separate top-level objects. Invariant: a Goal carries no plan,
no work package, no runtime, no procedure.

### 3.2 One Work Package schema (G3)

The **authoritative Work Package** is the runtime-contract structure of `05`:
`Identity` (id, parent goal, parent plan, priority), `Objective`, `Context`
(embedded Context Package), `Constraints`, `Resources`, `Skills`, `Inputs`,
`Outputs`, `Evidence`, `Completion Criteria`, `Status`, `Checkpoints`,
`Observability`. Planning's `04` fields are **mappings into** this one schema:
"required capabilities" → resolved into `Skills` + capability references;
"validation requirements" → `Evidence` + `Completion Criteria`. There is exactly
one Work Package definition; `04`/`05`/`14` all bind to it.

### 3.3 Execution Graph is a sibling artifact referenced by the Plan (G4)

The **Plan** holds the abstract approach (milestones, priorities, dependency
summary, rationale, **and a reference to its Execution Graph by id**). The
**Execution Graph** is a *separate, stateful artifact* (nodes, typed edges,
conditions, checkpoints, policies, state, metrics) with its own lifecycle,
consumed independently by Orchestration, Recovery, and Validation. Rationale:
the graph is stateful and separately evolving; nesting it inside the Plan object
couples two different lifecycles. The separate **"Dependency Graph" output is
eliminated** — dependencies are **edges in the Execution Graph**.

### 3.4 Observation is owned by Supervision; Execution emits Execution Events (G5)

**Execution emits raw `Execution Events`** (telemetry/progress/artifacts/
failures). **Supervision derives and owns the `Observation` object.** Execution
does **not** produce Observation objects. This gives one owner for Observation
and removes the dual-producer contradiction.

### 3.5 Intent Resolution is the canonical layer name (G6)

**"Intent Resolution"** (doc `16`, the precise specification) is canonical;
**"Executive Intelligence"** is deprecated as a layer name and retained only as a
historical alias in the glossary (AP-007). The Object Model ownership table reads
"Goal | Intent Resolution."

### 3.6 Validation: one enum, four results (G9)

A single **Validation Result** enum: `Passed`, `Failed`, `Partial`,
`RequiresReview`. The **Validation State** lifecycle (`Pending`,
`CollectingEvidence`, `Validating`, `Decided`, `Cancelled`) is separate from the
**result**; the terminal state `Decided` carries one of the four results.
"Waiting Human Review" is modeled as state `Validating` + result pending; the
final human outcome resolves to one of the four results. No two names for one
concept.

### 3.7 Artifact: one status vocabulary; Evidence by reference (G11)

The **Artifact** has one status lifecycle: `Draft → Generated → Validated →
Published → Archived` (the `Created/Produced/Versioned/Referenced` phrasing of
the Lifecycle section is descriptive narrative, not a second enum). The Artifact
**references** Evidence (by id); it does **not** embed an Evidence object.
Evidence remains a first-class object owned by Validation. Artifacts are
immutable-by-default (new version, never overwrite).

### 3.8 Evidence vs. Evidence Candidate

**Execution produces `Evidence Candidate`s; Validation promotes them to
`Evidence`.** Completion is determined from `Evidence`, never from runtime
self-report.

## 4. Alternatives Considered

- **Keep three Work Package definitions and reconcile in code.** *Rejected:*
  guarantees producer/consumer drift (the platform's highest contract risk).
- **Nest Execution Graph inside Plan (honor `02`).** *Rejected:* couples a
  stateful, independently-consumed topology to the Plan's lifecycle; Recovery and
  Orchestration need the graph as a first-class addressable artifact.
- **Let Execution own Observation (honor `08`).** *Rejected:* Supervision is the
  observer/health-deriver; Execution must stay "the smallest, most replaceable
  layer" and not own an evaluation-adjacent object.
- **Keep both layer names.** *Rejected:* two names for one layer invites two
  implementations.

## 5. Trade-offs

- **Gain:** exactly one schema per object; clean ownership; the seams (Context→
  Planning, Planning→Orchestration, Execution→Supervision→Validation) become
  contract-testable.
- **Cost:** the Object Model (`02`) and a few layer docs need addenda/glossary
  notes; some doc phrasings are retired.
- **Accepted because:** a single canonical object model is the precondition for
  Phase 1; ambiguity here multiplies across every layer.

## 6. Consequences

- **AP-101…AP-112** implement exactly these decisions; `contracts/` freezes them.
- **AP-007** records the Intent Resolution rename and fixes the index/glossary.
- The Object Model ownership table is updated: Goal | Intent Resolution;
  Observation | Supervision; Execution Graph | Planning (sibling of Plan).
- Contract tests (AP-006) enforce one-schema-per-object at every seam.

## 7. Risks

- **Touching the Object Model ripples everywhere.** *Mitigation:* done once, in
  Phase 0, before any schema work; frozen by `contracts/`.
- **Embedded Context Package bloat** in Work Packages (the chosen WP embeds
  Context). *Mitigation:* allow Context-by-reference for large packages (a
  versioning/relationship rule in `contracts/work_package.md`), tracked as
  sanctioned debt.

## 8. Validation Strategy

- **One-schema test:** each object resolves to a single schema; "defined twice"
  scan returns empty (AP-003 → AP-101).
- **Seam contract tests:** Goal→Context→Plan→WP→Execution→Validation each pass
  against the frozen contracts (AP-006).
- **Ownership test:** Observation produced only by Supervision; Evidence only by
  Validation; Execution emits only Events/Candidates.
- **Graph-reference test:** Plan references Execution Graph by id; no nested graph
  state inside Plan; no separate Dependency Graph object exists.

## 9. Migration Considerations

- **Nexus v1** has its own task/approval/event records. The v2 Work Package
  generalizes v1 tasks; map v1 task fields into the canonical WP during
  migration.
- v1 "Executive"/chat-planner concepts map to **Intent Resolution**; v1's
  server-stamped governance flags become Goal constraints + policy events.
- Migration is additive: introduce canonical objects alongside v1 records, then
  converge producers/consumers onto the frozen contracts.

## 10. Future Evolution

- Nested/hierarchical Work Packages, dynamic graph expansion, and reusable
  planning templates (deferred in the docs) extend these schemas without
  redefining them — versioning rules in each contract govern compatible growth.
- Goal metadata may grow (urgency models, multi-operator goals) within the Goal
  object's optional-field + versioning rules.
